The key question is not just when to read more, but also what prefix is safe enough to commit right now.

Project Story

The project is easiest to understand if you read it as two complementary answers to the same problem. DD + JS is a when to wait policy. Future / literal LCP is a what is safe to write policy. Together they frame premature commitment as a decision problem instead of a fixed wait-k schedule.

Main Line

Qwen direct vs Future / literal LCP

Keep the translator fixed at Qwen3-30B-A3B-Instruct-2507, then change only the commit rule.

  • Fair comparison: same translator, same wmt500 slice, same wait-k framing.
  • New method: sample English futures with Qwen3-4B-Base, translate each one, then commit only the Chinese literal LCP that survives quorum.
  • Interpretation: this directly targets safe-prefix commitment.
Supporting Line

NLLB wait-k vs DD / JS vs literal LCP

Keep the translator fixed at facebook/nllb-200-distilled-600M, then ask whether the same future-aware ideas still help on a smaller backbone.

  • Future LM only: Qwen3-4B-Base samples futures from the observed prefix only, never oracle source peeking.
  • Policy ideas: DD + JS decides when to wait; literal LCP decides what is safe to commit.
  • Interpretation: this is supporting evidence that the future-aware idea is not only a big-model trick.
Shared Pipeline

Speech still fits the same story

The speech demo is intentionally cascaded, not a separate research claim.

  • Audio front-end: openai/whisper-small produces streaming English text.
  • Same back-end agents: after ASR, the exact same DD and Future LCP logic applies.
  • Why this matters: it keeps the final project aligned with streaming speech translation without diluting the main MT contribution.

Method Map

The four methods below are grouped by the translator they actually evaluate. This makes the fairness story much clearer than showing everything as one long baseline list.

Main evidence · fixed Qwen translator

What changes in the Qwen line

The translator stays fixed. The only difference is whether you commit from one direct hypothesis or from a future-aware consensus over multiple hypotheses.

1Qwen direct

Qwen3-30B-A3B-Instruct-2507 translates the visible prefix directly. This is the fairness baseline for Method 2.

2Future / literal LCP

Qwen3-4B-Base samples English futures, the 30B model translates each future-conditioned string, and only the Chinese literal LCP beyond the committed prefix is allowed to write.

Supporting evidence · fixed NLLB translator

What changes in the NLLB line

The translator stays fixed at NLLB. The methods differ in either search strength or in how conservative the read / commit policy is when the future is uncertain.

1Baseline 1

Strict wait-k + greedy facebook/nllb-200-distilled-600M. No DD, no future-aware veto, no multi-hypothesis consensus.

2Baseline 2

Same wait-k policy, but the decoder uses NLLB beam search. This improves search, not commit safety.

3Method 1 · DD + JS

Qwen3-4B-Base samples English futures, each future induces an NLLB next-token distribution, and average pairwise Jensen-Shannon divergence decides whether to READ more or allow the commit.

4Method 2 · literal LCP

NLLB generates multiple candidate continuations, and only the literal Chinese prefix that survives cross-candidate agreement is committed. This is the small-model LCP variant.

Baseline 2 — Wait-k + facebook/nllb-200-distilled-600M beam search

Models / stack: same agent as Baseline 1, but beam_size > 1 (e.g. 8): decoding keeps multiple NLLB hypotheses and commits the next Chinese unit from the best beam. This demo uses the same READ/WRITE toy steps as Baseline 1—only the decode captions differ. In real runs, --uncertainty-threshold is separate: a huge value (plus --max-extra-reads 0) turns off STTR read-more; a finite value lets _translate_with_gate issue extra READs— that is not a per-step “beam gate” in SimulEval.

What beam_size = 8 is doing (not “top‑8 full sentences”)

Inside NLLB decoding, beam search keeps the 8 best partial target hypotheses (equal decode depth at a given step), ranked by cumulative score / log‑prob. Each step: extend each surviving beam with candidate next subwords (or chars), rescore, then prune back to 8. Greedy (beam_size = 1) is the case where only one partial hypothesis survives. This panel is illustrative—scores are fake; real behaviour is in HF generate / your NLLB config.

Toy snapshot: English prefix fixed as The river was (same sentence as the walkthrough). The green row wins; on a WRITE step the agent commits that winning extension’s new piece (here the character 以, matching 这条河以他堂兄…).

With beam_size = 1, only the single best greedy extension survives each step. With beam_size = 8, up to eight partial hypotheses stay alive between prune steps; the policy still emits one next target unit per SimulEval WRITE from the current best beam (implementation detail: full hypothesis vs one char depends on tokenizer—this demo uses one Chinese character for clarity).

🔑 Key caching rule (applies to both Baseline 1 & 2) As long as the source prefix has not changed, every consecutive WRITE takes the next character directly from the cached beam-search result — beam search is NOT re-run. The beam search only re-runs when a new READ arrives and the source prefix grows.

In code (sttr_enzh_agent.py → _translate_with_gate):
  • Cache key = (source_text, source_finished)
  • Cache hit (source unchanged) → return _current_translation[tgt_len] directly, no NLLB call
  • Cache miss (new word arrived) → re-run _draft_with_uncertainty → full beam decode → store new _current_translation

Practical consequence: beam search is called roughly once per READ, not once per WRITE. A sentence with 8 source words and wait-k = 3 typically triggers beam search ≈ 5–8 times (one per new word), while WRITE steps just index into the cached sequence.
